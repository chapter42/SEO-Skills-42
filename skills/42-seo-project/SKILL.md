# SEO Project Orchestrator

Guides the complete SEO/GEO improvement lifecycle for a website. Manages state
across sessions, decides which skills to run when, and tracks progress from
first audit to ongoing monitoring.

**This skill does NOT duplicate skill logic.** It delegates to existing skills
and adds sequencing, state tracking, and intelligent routing on top.

---

## Quick Reference

| Command | What it does |
|---------|-------------|
| `/seo-project init` | Interactive setup: URLs, API keys, export dir, .env |
| `/seo-project start <url>` | Quick start: init + crawl + discover in one go |
| `/seo-project next` | Auto-detect current phase, execute next logical step |
| `/seo-project status` | Show progress dashboard without taking action |
| `/seo-project discover [url]` | Run/re-run Phase 1: baseline audit |
| `/seo-project diagnose` | Run Phase 2: deep-dive on weak categories |
| `/seo-project strategize` | Phase 3: interactive strategy session, lock decisions |
| `/seo-project fix [wave]` | Phase 4: execute fixes, optionally specific wave (1-4) |
| `/seo-project verify` | Phase 5: re-audit and compare to baseline |
| `/seo-project monitor` | Phase 6: monthly re-check cycle |
| `/seo-project skip <phase>` | Skip a phase, advance to next |
| `/seo-project history` | Show audit history and score trends |
| `/seo-project reset` | Clear state, start over (asks confirmation) |

## Phases

The SEO improvement lifecycle has an init step + 7 phases.

```
INIT ──→ 0. CRAWL ──→ 1. DISCOVER ──→ 2. DIAGNOSE ──→ 3. STRATEGIZE ──→ 4. IMPLEMENT ──→ 5. VERIFY ──→ 6. MONITOR
(setup)    (SF data)    (baseline)     (deep-dive)     (lock decisions)   (fix in waves)   (re-audit)    (ongoing)
```

### INIT — Project Setup

**Goal:** Gather all project info, set up directories, configure API keys,
and prepare the environment before any crawling or analysis begins.

**This is interactive.** Ask the user each question, confirm, then proceed.

**Step 1: Project Details**

Ask the user:
1. **Primary URL** — The homepage URL to audit (e.g., `https://example.com`)
2. **Additional URLs** (optional) — Specific pages to analyze in detail
3. **Business type** — SaaS, e-commerce, local service, publisher, agency, or auto-detect?
4. **Target market/language** — e.g., NL, EN, DE, multi-language
5. **Competitors** (optional) — Up to 3 competitor URLs for comparison

**Step 2: Directory Setup**

Create the project directory structure:

```
.seo-project/
└── <domain>/
    ├── STATE.md
    ├── exports/              # Screaming Frog CSV exports land here
    │   └── .gitkeep
    ├── reports/              # Generated reports
    │   └── .gitkeep
    └── history/              # Audit snapshots over time
        └── .gitkeep
```

**Step 3: Environment & API Keys**

Check which tools/APIs are available and create `.seo-project/<domain>/.env`:

```
# SEO Project: <domain>
# Generated by /seo-project init on <date>

# === Required ===
SITE_URL=https://example.com
DOMAIN=example-com
BUSINESS_TYPE=saas
LANGUAGE=nl

# === Screaming Frog ===
SF_AVAILABLE=true                    # auto-detected
SF_EXPORT_DIR=.seo-project/<domain>/exports
# SF_CONFIG=/path/to/custom.seospider  # optional: custom crawl profile

# === Data APIs (optional — skills degrade gracefully without these) ===
DATAFORSEO_LOGIN=                    # ~$0.002/query — https://dataforseo.com
DATAFORSEO_PASSWORD=
GSC_SERVICE_ACCOUNT_PATH=            # Google Search Console service account JSON
AHREFS_API_KEY=                      # Or use Ahrefs MCP server
SEMRUSH_API_KEY=                     # Or use SEMRush MCP server

# === Embedding Provider (for keyword-mapper, passage-analyzer, near-duplicates) ===
GOOGLE_API_KEY=                      # Gemini embeddings (keyword-mapper, passage-analyzer)
```

**Detection logic:**
1. Check if Screaming Frog CLI is available: `which screamingfrogseospider` or check common install paths
2. Check for existing `.env` files: `~/.config/seo-agi/.env`, project-level `.env`
3. If keys already exist in env or config files, pre-fill them (don't ask again)
4. For missing optional keys, show what they unlock but don't block progress

**Step 4: Capability Summary**

Show what's available and what's limited:

```
SEO Project Setup Complete: example.com

Tools detected:
  ✓ Screaming Frog (v21.2, licensed)
  ✓ DataForSEO (credentials found)
  ✓ Google Search Console (service account configured)
  ✗ Ahrefs API (not configured — competitive analysis will use WebSearch fallback)
  ✗ Embedding provider (not configured — keyword-mapper and passage-analyzer will use heuristic mode)

Project directory: .seo-project/example-com/
Export directory:  .seo-project/example-com/exports/

Ready to crawl. Run /seo-project next to start Phase 0 (Screaming Frog crawl).
```

**Step 5: Update STATE.md**

Set `init: completed` with detected capabilities.

### Phase 0: CRAWL — Data Foundation

**Goal:** Run a Screaming Frog crawl to create the data foundation that most
downstream skills depend on.

**Delegates to:**
- `/42:screaming-frog crawl <url>` → Full technical crawl
- `/42:screaming-frog export <url> --bulk` → Export all tabs as CSV

**Why first:** Many 42-skills require Screaming Frog export data as input:
- `42-keyword-mapper` → needs SF embedding export
- `42-link-graph` → needs Inlinks:All export
- `42-passage-analyzer` → needs SF text export
- `42-migration` → needs Internal:HTML export
- `42-meta-optimizer` → needs Internal:HTML export
- `42-striking-distance` → needs Internal:HTML export
- `42-page-health` → needs crawl data for composite scoring
- `42-near-duplicates` → needs content hashes from SF
- `42-readability` → needs SF text export

**Actions:**
1. Create `.seo-project/<domain>/` directory
2. Copy `templates/STATE.md` → initialize with site URL, domain, and current date
3. Check Screaming Frog availability: `/42:screaming-frog check`
4. If SF available:
   - Run crawl: `/42:screaming-frog crawl <url>`
   - Export all tabs: `/42:screaming-frog export <url> --bulk`
   - Record crawl path in STATE.md
5. If SF NOT available:
   - Log warning: "Screaming Frog not installed — some 42-skills will have limited data"
   - Mark phase as `skipped` (not `completed`)
   - Proceed to Phase 1 with WebFetch-based crawling only
6. Update STATE.md: `crawl: completed`, record export paths

**Output:** Screaming Frog export CSVs stored in SF's default output directory.
Path recorded in `.seo-project/<domain>/STATE.md` under `crawl.export_path`.

**Note:** This phase is optional but strongly recommended. The orchestrator
proceeds without SF data, but many Phase 2 and Phase 4 skills will have
reduced capabilities.

### Phase 1: DISCOVER — Baseline

**Goal:** Understand the site, measure current scores, establish baseline.

**Delegates to:**
- `/seo audit <url>` → SEO Health Score (0-100)
- `/geo audit <url>` → GEO Score (0-100)

**Uses Phase 0 data:** If SF crawl data is available, the audit skills can
reference it for richer technical analysis instead of re-crawling via WebFetch.

**Actions:**
1. Run `/seo audit` and `/geo audit` (can run in parallel via subagents)
2. Capture scores per category in `DISCOVERY.md`
3. Update STATE.md: `discover: completed`, record scores

**Output:** `.seo-project/<domain>/DISCOVERY.md`

---

### Phase 2: DIAGNOSE — Deep-Dive on Weak Areas

**Goal:** Identify specific problems and prioritize by impact.

**Conditional delegation based on Phase 1 scores:**

| Condition | Skill(s) to run |
|-----------|-----------------|
| Technical < 70 | `/42:technical <url>` |
| Content < 70 | `/42:content <url>` + `/42:content --geo <url>` |
| Schema < 70 | `/42:structured-data --seo <url>` + `/42:structured-data --geo <url>` |
| AI Crawlers blocked | `/42:crawlers <url>` |
| Citability < 50 | `/42:citability <url>` |
| Images < 60 | `/42:images <url>` |
| Sitemap issues | `/42:sitemap <url>` |
| Hreflang present but broken | `/42:hreflang <url>` |
| Keyword cannibalization suspected | `/42:cannibalization` |
| E-commerce site | `/42:ecom-taxonomy` |
| Quality Rater deep-dive needed | `/42:qrg <url>` |

**Actions:**
1. Read DISCOVERY.md scores
2. Determine which categories need deep-dive (threshold-based)
3. Run relevant specialist skills (parallel where possible)
4. Synthesize findings into prioritized issue list (Critical/High/Medium/Low)
5. Update STATE.md: `diagnose: completed`, record issue counts

**Output:** `.seo-project/<domain>/DIAGNOSIS.md`

---

### Phase 3: STRATEGIZE — Lock Decisions

**Goal:** Decide what to fix, in what order, with what constraints.

**Delegates to:**
- `/seo-plan` — strategic planning with industry templates
- `/42:competitor-pages` — competitive gap analysis (optional)
- `/42:topical-map` — topical authority mapping (optional)
- `/42:serp-cluster` — SERP clustering for content planning (optional)
- `/42:audience-angles` — content angles via QPAFFCGMIM framework (optional)
- `/42:paa-scraper` — People Also Ask data for content gaps (optional)
- Interactive session with user

**Actions:**
1. Present DIAGNOSIS.md summary to user
2. Ask user to confirm/adjust priorities
3. Lock decisions with IDs (D-01, D-02, etc.):
   - Scope: which issues to address now vs later
   - Priority order within scope
   - Budget/timeline constraints
   - Content language and target market
   - Tools/APIs available (DataForSEO, Ahrefs, GSC, etc.)
4. Generate implementation waves based on locked decisions
5. Update STATE.md: `strategize: completed`

**Output:** `.seo-project/<domain>/STRATEGY.md`

**Important:** Once decisions are locked, they persist across sessions. The
orchestrator references STRATEGY.md before every implementation action to
prevent drift from agreed-upon priorities.

---

### Phase 4: IMPLEMENT — Fix in Priority Waves

**Goal:** Execute fixes in order of impact, tracking completion per task.

Four waves, each containing related tasks:

| Wave | Priority | Typical skills |
|------|----------|---------------|
| 1 | Critical | `42:technical` fixes, `42:crawlers` unblocking, `42:structured-data` generation, broken canonical/indexing fixes |
| 2 | High | `42:content` rewrites, `42:llmstxt` generation, `42:genai-optimizer` passage rewrites, `42:meta-optimizer` bulk title/meta, `42:sitemap` generation |
| 3 | Medium | `42:images` optimization, `42:internal-links` improvements, `42:programmatic-seo` pages, `42:title-optimizer` headlines, `42:product-titles` (e-commerce), `42:blog-schema` generation |
| 4 | Low | `42:blog-seo-check` validation, `42:blog-geo` optimization, `42:hreflang` implementation, `42:content-repurposer` hergebruik, advanced GEO tweaks |

**Actions per wave:**
1. Read STRATEGY.md for locked decisions
2. List tasks for this wave from PROGRESS.md
3. Execute each task by delegating to the relevant skill
4. Mark task as completed in PROGRESS.md
5. After all tasks in wave: update STATE.md wave counter
6. Ask user: proceed to next wave or pause?

**Output:** `.seo-project/<domain>/PROGRESS.md` (updated incrementally)

**Note:** Wave contents are determined by STRATEGY.md, not hardcoded. The table
above shows typical assignments. The strategize phase may move tasks between
waves based on user priorities.

---

### Phase 5: VERIFY — Re-Audit and Compare

**Goal:** Confirm fixes worked by comparing to baseline.

**Delegates to:**
- Re-run `/seo audit` + `/geo audit`
- Apply `/geo-compare` delta pattern

**Actions:**
1. Run fresh audit (same skills as Phase 1)
2. Compare every score to DISCOVERY.md baseline
3. Calculate deltas per category
4. Highlight: biggest wins, remaining gaps, regressions
5. Generate before/after report
6. Update STATE.md: `verify: completed`

**Output:** `.seo-project/<domain>/VERIFICATION.md`

---

### Phase 6: MONITOR — Ongoing Optimization

**Goal:** Keep scores up, detect decay, find new opportunities.

**Delegates to (per check cycle):**
- `/seo audit` + `/geo audit` — re-audit for score comparison
- `/geo-compare` — delta tracking
- `/42:content-decay` — detect pages losing traffic/rankings
- `/42:share-of-voice` — track market share in search results
- `/42:striking-distance` — find new quick-win keyword opportunities

**Actions:**
1. Record verification snapshot in `history/audit-<date>.md`
2. Set next check date (default: 30 days)
3. On next check:
   - Run discovery-level audit
   - Compare to most recent snapshot
   - Run `42:content-decay` to detect declining pages
   - Run `42:share-of-voice` to track competitive position
   - Flag regressions and new opportunities
   - Update MONITOR.md with trend data
4. Optionally trigger new diagnose→fix cycle for regressions

**Output:** `.seo-project/<domain>/MONITOR.md` + `history/` snapshots

## State Machine — `/seo-project next` Routing

```
Read .seo-project/<domain>/STATE.md
  |
  +-- No .seo-project/ exists
  |   -> "No project found. Run: /seo-project init or /seo-project start <url>"
  |
  +-- init: pending
  |   -> Run INIT (interactive setup)
  |
  +-- init: completed, crawl: pending
  |   -> Run Phase 0 (Screaming Frog crawl)
  |
  +-- crawl: completed OR skipped, discover: pending
  |   -> Run Phase 1 (discover)
  |
  +-- discover: in_progress
  |   -> Resume Phase 1 (check what's missing)
  |
  +-- discover: completed, diagnose: pending
  |   -> Show discovery summary, then run Phase 2
  |
  +-- diagnose: completed, strategize: pending
  |   -> Show diagnosis summary, then run Phase 3
  |
  +-- strategize: completed, implement: pending
  |   -> Show strategy summary, begin wave 1
  |
  +-- implement: in_progress, wave N < total
  |   -> Continue current wave or start next wave
  |
  +-- implement: all waves completed
  |   -> Run Phase 5 (verify)
  |
  +-- verify: completed, monitor: pending
  |   -> Show verification results, set up monitoring
  |
  +-- monitor: active
  |   -> Check if re-audit is due
  |   -> If due: run re-audit cycle
  |   -> If not: show "Next check: <date>. Score trend: ..."
```

## `/seo-project start <url>` — Initialization

**`/seo-project start <url>`** is a shortcut that runs INIT + Phase 0 + Phase 1 in sequence:
1. Run INIT with the provided URL (ask remaining questions interactively)
2. Immediately proceed to Phase 0 (crawl)
3. Then Phase 1 (discover)

**`/seo-project init`** runs only the setup, without starting any analysis.
Use this when you want to configure everything first and start later.

## Decision Locking (Phase 3)

Decisions are captured with IDs for traceability:

```markdown
## Locked Decisions

- **D-01**: Focus on Dutch market only (no hreflang needed)
- **D-02**: Prioritize GEO over traditional SEO (AI traffic growing 527%)
- **D-03**: Budget: 20 hours total, spread across 4 waves
- **D-04**: DataForSEO available for keyword research
- **D-05**: Blog content rewrites before new content creation
- **D-06**: No programmatic pages (site too small)
```

All subsequent phases reference these decisions. If a task would contradict
a locked decision, flag it and ask the user before proceeding.

This skill sits ABOVE the existing `seo/` and `geo/` orchestrators:

```
seo-project (project lifecycle)
  |
  +-- 42:audit (unified SEO+GEO audit orchestrator)
  |   +-- 42:technical, 42:content, 42:structured-data, ...
  |   +-- 42:citability, 42:crawlers, 42:brand-mentions, ...
```

**All skills remain independently invocable.** This orchestrator adds
project-level state and sequencing on top, but never locks skills behind phases.
Users can always run any skill directly without a project context.

## Session Continuity

All state lives in `.seo-project/<domain>/` files. When a conversation resets:

1. `/seo-project next` reads STATE.md to determine position
2. Reads the relevant phase file (DISCOVERY.md, PROGRESS.md, etc.)
3. Resumes from exactly where the previous session left off
4. No context is lost between sessions

---

## Error Handling

- **Skill not available**: If a delegated skill fails to load, log the error
  in STATE.md `blockers` field and suggest the user run it manually
- **Audit timeout**: Large sites may take long. If interrupted, STATE.md
  records partial progress so `/seo-project next` can resume
- **Score regression in verify**: Flag regressions prominently and suggest
  a targeted diagnose→fix mini-cycle for regressed categories only

## Quality Gates (Inherited)

This orchestrator inherits all quality gates from sub-skills:
- Max 500 pages per crawl (seo-audit)
- Max 50 pages per GEO audit (geo-audit)
- Never recommend HowTo schema (deprecated Sept 2023)
- FAQ schema restrictions (Aug 2023)
- All CWV references use INP, never FID
- Respect robots.txt always
